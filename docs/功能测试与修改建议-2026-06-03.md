## 学生画像核查

结论：学生画像功能真实存在，并且是长期持久化，不是仅内存短期状态。

数据落点：

- 主画像：`student_profiles.profile_json`
- 诊断历史：`diagnostic_results`
- 认知画像：`cognitive_profiles`
- 学习偏好：`learning_preferences`

可用接口：

- `GET /api/diagnostic/profile`
- `POST /api/diagnostic/quick`
- `POST /api/diagnostic/submit`
- `GET /api/diagnostic/result`
- `GET /api/diagnostic/history`
- `POST /api/app/diagnosis/submit`
- `GET /api/app/intelligence`

实测结果：

- `POST /api/diagnostic/quick` 可根据自然语言生成画像，并写入 `student_profiles`。
- `GET /api/diagnostic/profile` 可读取长期画像。
- 使用 Unicode 测试后，中文可正常保存：例如 `freeText` 保留“我是软件工程学生，喜欢图解和项目实践...”。
- `POST /api/app/diagnosis/submit` 写入的是 `cognitive_profiles` 和 `learning_preferences`，不是 `student_profiles`。

## 个性化路径核查

结论：路径接口存在，但“根据学生画像个性化”的实现不完整。

可用接口：

- `POST /api/ai/learning-path`
- `GET /api/ai/today-card`
- `GET /api/ai/learning-list`
- `POST /api/ai/learning-list`

实测结果：

- `POST /api/ai/learning-path` 返回成功。
- 返回路径使用 `knowledge_nodes` 和 `questions` 计算掌握度。
- 代码声称读取 `student_profiles.profile_json`，但实际只识别 `learningStyle`、`learning_style`、`overview.primaryStyle` 等字段。
- 诊断画像实际写入的是 `cognitiveStyle.type`、`learningPatterns.注意力持续时间`、`multimodalPreferences` 等字段，路径生成器没有读取这些字段。
- 当前返回给前端的数据没有暴露 `profileContext`，用户无法确认路径是如何被画像影响的。
- 实测视觉型画像后，路径资源仍优先返回 document/document/quiz/lab，说明资源偏好落入默认读写型权重。

失败接口：

- `GET /api/ai/today-card` 返回 500。
- `GET /api/ai/learning-list` 返回 500。

失败原因：

- `today-card` 查询 `questions.content/options/node_id/score`，但实际表字段是 `question/options_json/knowledge_id` 等。
- `learning-list` 查询 `knowledge_nodes.node_name`，但实际字段是 `name`。

## P0 修改建议

1. 统一数据库 schema 与代码字段

   - `questions`: 统一为 `question/options_json/knowledge_id`，或补兼容视图/迁移字段 `content/options/node_id/score`。
   - `knowledge_nodes`: 补 `content_url/bvid/video_platform`，或代码改为只查真实字段。
   - `knowledge_nodes.node_name` 改为 `knowledge_nodes.name`。
   - `users`: 补 `phone/gender/birthday/bio/avatar`，或认证接口降级查询。
   - 修复 `user_answers.user_answer` 与实际字段不一致问题。

2. 修复画像到路径的真实个性化

   - `AIPathGenerator.getStyleWeights()` 应读取 `profile.cognitiveStyle.type` 和 `profile.multimodalPreferences`。
   - `getDailyMinutes()` 应读取诊断结果的 `analysis.summary.dailyStudyMinutes`，或把 `dailyTimeMinutes` 写入 `student_profiles.profile_json`。
   - `getAttentionSpan()` 应读取 `learningPatterns.注意力持续时间`。
   - `/api/ai/learning-path` 返回中应保留 `profileContext`，用于前端展示“为什么这样推荐”。

3. 统一画像体系

   - 明确 `student_profiles` 是长期画像主表。
   - `cognitive_profiles` 是认知诊断快照。
   - `learning_preferences` 是偏好证据表。
   - `POST /api/app/diagnosis/submit` 应同步更新或关联 `student_profiles`，否则两套画像割裂。

4. 修复路径掌握度来源

   - 当前路径使用 `knowledge_nodes`，答题表使用 `questions.knowledge_id` 指向 `knowledge_points`，存在 ID 语义错配风险。
   - 建议建立 `knowledge_nodes` 与 `knowledge_points` 的映射，或统一只使用一套知识点表。

5. 修复 RAG 检索可用性

   - 前端和文档统一使用 `/api/rag/query`、`/api/rag/ask`。
   - 优化中文分词与字段匹配，至少支持 `course/title/knowledge_point/chunk_text` 的 LIKE 回退。
   - 查询无命中时返回可解释原因和推荐关键词。

## P1 修改建议

1. 为关键 API 加集成测试

   - 登录、画像生成、画像读取、路径生成、今日卡片、学习列表、RAG 查询、导师消息、笔记卡片、教师路径。

2. 增加启动时 schema 自检

   - 服务启动时检查关键列是否存在，缺失时给出明确错误。
   - 不建议 API 请求时才暴露 `Unknown column`。

3. 增加画像版本与证据日志

   - 每次画像更新保存来源：问卷、自然语言、答题、导师对话、教师干预。
   - 支持查看画像变化历史，而不是只覆盖最新 JSON。

4. 前端显示个性化依据

   - 在路径卡片中展示：学习风格、可用时间、专注时长、薄弱点、推荐原因。
   - 用户能判断“这是按我的画像生成的”，而不是只看到普通路径。

5. 外部能力配置提示

   - 对 XFYUN/TTS/虚拟人等功能，前端应显示“未配置/不可用”，避免用户误以为功能坏了。

## 总结

学生画像是存在的，并且有长期持久化；但目前“画像驱动个人路径”的链路没有完全打通。主要问题不是概念缺失，而是 schema 漂移、两套画像数据割裂、路径生成器读取字段不匹配。优先修复 P0 后，再补集成测试，才能支撑你后续完整功能验收。
